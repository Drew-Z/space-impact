# Post-Release Backlog

## 文档目的

这份文档用于承接 `v1.1.1` 发布后的维护与迭代工作，把“还可以继续优化什么”整理成清晰的后续待办，而不是在没有优先级的情况下继续扩散开发。

当前项目已经完成可发布版本，因此后续原则是：

- 先修体验尖刺，再补内容表现
- 先做低风险、小步快跑的更新，再考虑中等规模增强
- 保持《Space Impact》初代风格，不偏向复杂现代系统

## 当前已落地的预处理

在 `v1.1.1` 发布后，已先完成两项低风险收口，为后续版本铺路：

- 把敌人基础配置、Boss 支援覆盖规则与 Boss 文案映射抽到 `scripts/game/stage_catalog.gd`
- 把关卡时间表抽到 `scripts/game/stage_schedule.gd`，把掉落 / 警报等运行时平衡规则抽到 `scripts/game/run_balance.gd`
- 新增 `skirmisher`、`lancer` 两类后段敌人，用于加强 Sector 4-5 与 Boss 支援波次差异

## 当前版本基线

`v1.1.1` 已具备以下完整结构：

- 主菜单、阶段继续、设置、语言切换
- 5 个常规 Sector 与最终 Boss 阶段
- 13 级武器成长与高阶持续主炮
- 单命 `HULL 3` 与“先掉武器等级、后掉船体”的受伤规则
- Boss 支援小怪、结算流程、最高分记录
- 主菜单与战斗区分音乐，主要战斗音效已完成一轮打磨
- Windows 导出版已完成实际验证与发布

## 优先级 A：`v1.1.2` 热修与体验微调

这些工作适合继续保持当前结构不变，以小版本修复和微调为主。

### 1. Boss 战时长微调

目标：

- 继续确认前两关 Boss 是否存在“略拖”的情况
- 保持后段 Boss 和最终 Boss 的持久压迫感

建议动作：

- 重点观察 Sector 1 与 Sector 2 的 Boss 血量体感
- 如有必要，只小幅下调前段 Boss 耐久，不动后段结构

### 2. 音效层级最后一轮混音

目标：

- 让警报、激光、普通击败、碰撞和拾取的音量关系更稳定
- 避免在长时间游玩中出现单个音效过于抢戏

建议动作：

- 逐项记录主观听感
- 只做音量和包络微调，避免再大改音色结构

### 3. 背景与战斗可读性复查

目标：

- 确保不同关卡背景有差异，但不压过敌机、敌弹和掉落物

建议动作：

- 重点复查 Sector 4、Sector 5、Final Core
- 如果背景过抢，只降低亮度、线条密度或滚动强度

### 4. 导出版整体验证回归

目标：

- 以导出的 `SpaceWar.exe` 为准，确保发布包和编辑器版本体验一致

建议动作：

- 再跑一次完整流程
- 重点验证设置页导航、阶段继续、最高分持久化和最终结算

## 优先级 B：`v1.2.0` 内容增强

这些内容适合做成一个更明确的新版本，而不是混进热修。

### 1. 敌机外形与行为差异增强

目标：

- 让每关普通敌机在轮廓和行为上更有识别度

建议动作：

- 为现有敌机增加更清楚的机体差异
- 在不增加复杂系统的前提下，补充少量新攻击模式

### 2. Boss 专属表现强化

目标：

- 让每个 Boss 不只是在参数上不同，而是在战斗记忆点上更鲜明

建议动作：

- 为每个 Boss 再补一项代表性机制或明显阶段表现
- 优先增强视觉识别和攻击节奏，不先堆更多系统

### 3. 反馈演出升级

目标：

- 进一步提高成品感与展示价值

建议动作：

- 优化爆炸层次、受击闪烁、拾取反馈
- 给关键节点补充更完整的提示和演出节奏

### 4. 资源替换与风格统一

目标：

- 在不偏离初代 Space Impact 气质的前提下，减少占位感

建议动作：

- 补更稳定的 UI 图形资源
- 优化字体、边框、图标与背景风格统一性

## 优先级 C：展示与维护

这些工作不一定改变玩法，但能明显提升项目展示和后续维护效率。

### 1. README 展示升级

目标：

- 让仓库首页能快速说明项目定位、当前版本和运行方式

建议动作：

- 增加当前版本介绍
- 增加截图、导出说明和发布链接

### 2. 演示素材整理

目标：

- 为发布页、简历、作品集或社交平台准备稳定素材

建议动作：

- 录制一段短视频或 GIF
- 选取主菜单、常规战斗、Boss 战、结算页截图

### 3. 反馈收集机制

目标：

- 用真实试玩反馈驱动下一版优化，而不是凭感觉继续扩展

建议动作：

- 记录玩家首次游玩卡点
- 记录对难度、音效、可读性和手感的主观反馈

## 建议执行顺序

建议按以下顺序推进：

1. 先做一轮 `v1.1.2` 热修候选梳理
2. 从中选择 3 到 5 个最小必要项
3. 完成一次小版本修正并重新验证导出版
4. 再启动 `v1.2.0` 的内容增强计划

## 当前结论

当前项目已经不需要再做大范围基础建设。

最推荐的下一步是：

- 先把 `v1.1.2` 作为“发布后体验收口”版本
- 只修真正影响体验的细节项
- 等反馈稳定后，再进入更明显的内容增强阶段
